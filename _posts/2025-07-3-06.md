There is a point in learning to program when the way you think about a problem shifts fundamentally. Not the way you write code — the way you think about the problem itself, before a single line is written. For me in Section A, that shift arrived during the week Programming Fundamentals spent on functions, and I did not fully realize it was happening until I looked back at the code I had written the week before and found it almost unreadable.

Before this week, I wrote programs the same way I might write a shopping list: everything in sequence, from top to bottom, one thing after another until it was done. The programs ran. They usually produced correct output. But rereading them was uncomfortable, because everything was tangled together and there was no clear separation between the part of the code that did one thing and the part that did something else. Changing anything required understanding everything, because everything was connected to everything else in one long unbroken sequence.

Functions changed this completely. A function is a named, self-contained piece of logic. You define what it needs as input, what it promises to produce as output, and what it does internally to get from one to the other. Once you have defined it, you call it by name whenever you need it, and you do not need to think about its internal workings every time you do. The program stops being a single long sequence and becomes a collection of named, understandable pieces that work together — each piece small enough to understand in isolation, the whole thing comprehensible because the pieces are.

The first assignment that truly tested this was one that asked for several string operations — checking whether a string was a palindrome, counting its vowels, reversing it — and then combining the results into a formatted report. Several of my classmates wrote all of this in one continuous block, which meant their code was long, repetitive, and difficult to change. My approach, after this week's lectures had started to sink in, was to write a separate function for each operation and then call them in the reporting section. The program that resulted was shorter overall, and each individual function was simple enough to read and understand in about thirty seconds. When I discovered a bug in my vowel counter, I fixed it in one place and the fix applied everywhere that function was used. One of my classmates spent twenty minutes tracking down the same bug across three separate places in their single-block solution.

The idea of a function contract — this is what I take in, this is what I give back — turned out to be one of the most practically important things in this whole unit. I remember being confused early on about the difference between a function that modifies something it is given and a function that returns a new value without touching anything outside itself. This confusion produced several bugs where I expected a function to have changed a value and it had not, because what it had actually done was return a new value that I then failed to store anywhere. Reading function signatures much more carefully, understanding what each function takes and what it returns before writing any code that calls it, became one of the most reliable bug-prevention habits of the semester.

The broader habit of decomposition — breaking a large problem into smaller named pieces before writing any code — started forming during this week and has never left. When I get a new assignment now, the first thing I do on paper is ask: what are the distinct sub-problems here? Which of those sub-problems is small and clear enough to be a single function? Which needs to be broken down further? This planning step takes time, but it has paid back that time many times over in reduced debugging, cleaner code, and programs that I can actually come back to a week later and still understand.
One thing this week made visible that I had not anticipated: the habit of function-first thinking changed how I collaborated with classmates during Section A lab sessions. When a group exercise asked two people to split work, having well-defined functions with clear input and output contracts made it possible to divide the work cleanly — you write this function, I will write that one, and we can test them separately before combining. That collaborative benefit of modularity, which mirrors almost exactly how software teams divide work in professional settings, was not something I had thought about before this week. It is one of the things I suspect was built into the course design deliberately, as preparation for exactly the kind of team-based project work the semester would eventually demand.


Looking back at this week now, I can see how directly the decomposition habit carried over into the database work running in parallel. The impulse to break a complicated real-world structure into small, well-defined, single-responsibility pieces — which is exactly what functions represent in programming — is also exactly what normalization asks you to do to a database schema. A good schema breaks the world into tables, each one focused on one kind of thing, the same way a good program breaks the logic into functions, each one focused on one task. I did not see that connection at the time. It only became clear weeks later, when the heart disease project brought both disciplines together in a single piece of work, and I found myself applying the same instincts I had learned in PF to decisions about table structure in the database. That connection, running underneath two apparently separate courses, is one of the most valuable things [Dr. Bilal Ahmad](https://www.facebook.com/Dr.BilalAhm) built into this semester.
