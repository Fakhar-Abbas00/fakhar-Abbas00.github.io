After several weeks of working with database design entirely on paper — drawing ER diagrams, working through normalization rules, arguing about table structure — there was something genuinely satisfying about finally writing a command that a real database engine would actually execute. SQL arrived in early June, and it was the moment Database Systems shifted from being a course about thinking carefully about design to being a course about putting that design to work in something real. The first time I ran a CREATE TABLE statement and watched a table appear in the database, I felt something I had not expected: a small but real sense of ownership over the structure that had spent weeks living only in my notes.

SQL is divided into two broad categories of commands that serve very different purposes. The first is DDL, or data definition language — the commands that create and shape the structure of the database itself. CREATE TABLE, ALTER TABLE, DROP TABLE. These are the commands that turn the diagrams from the normalization unit into real, enforced structures. Writing my first CREATE TABLE statement felt almost anticlimactic after the conceptual depth of what had come before, because the translation from a normalized design to SQL was mostly mechanical. If the design was solid, the SQL was straightforward. If the SQL was difficult to write, that was usually a signal that the design still had problems.

What Dr. Bilal Ahmad emphasized most strongly in this unit was constraints. A constraint is not just a structural detail — it is a rule that the database will enforce automatically, on every insert and every update, without any programmer needing to remember to check it in code. A NOT NULL constraint on a field means the database will simply reject any attempt to leave that field empty. A UNIQUE constraint means the database will reject any attempt to store a duplicate value where only one should exist. A foreign key constraint means the database will reject any attempt to create a reference to a row that does not exist in the referenced table. These constraints are how the careful thinking done during the design phase becomes something the system enforces rather than something every programmer has to manually implement every time they touch the data.

DML — data manipulation language — came next, and this is where SQL starts to feel like a tool rather than a formality. INSERT, UPDATE, DELETE, and SELECT are the commands for actually working with data rather than just defining where it will live. The first time we deliberately tried to insert a row that violated one of our own constraints and watched the database reject it, rather than silently accepting bad data the way a poorly designed system would, was a small but satisfying demonstration that the design work from previous weeks had actually produced something that protected itself.

SELECT queries are where most of the intellectual work in SQL lives, and they are also where most of the initial confusion lives. Simple queries with WHERE clauses were easy enough, but JOIN operations introduced a category of subtlety I had not anticipated. A JOIN combines information from multiple tables based on a matching condition, but the type of JOIN determines what happens when a row in one table has no match in the other. An inner join silently excludes any row without a match on both sides. I discovered this the hard way when a query I was sure was correct returned a result set that looked plausible but was missing exactly the rows I cared most about — the ones that had no matching entry on the other side of the join. Understanding this, and learning to choose between inner and outer joins deliberately rather than by default, was one of the more important moments of this unit.

Aggregate functions and subqueries were where SQL started to feel genuinely powerful. The ability to ask a database not just for raw data but for computed summaries — count how many, find the average, group by category — pushed SQL from a retrieval language toward something capable of answering real analytical questions. Writing a query that used a subquery to find every record whose value exceeded the average of the whole table — in a single statement — was the first moment SQL felt to me like a tool I would actually want to use for something meaningful rather than as a course exercise.

The habit Dr. Ahmad consistently encouraged during this unit was one that connected directly to lessons from PF: before writing a query, state in plain language exactly what question you are trying to answer. More than once during this unit, a query that seemed impossible to write turned out to be simple once I had stopped wrestling with syntax and forced myself to express the question in a single clear sentence. That habit — question first, code second — is the same discipline that PF had been building under the name of writing out steps before coding. It simply wore different clothes here.

The SQL we learned in this unit fed directly into the heart disease prediction project that our group was already beginning to plan. The queries we would later write to retrieve patient records, filter by clinical features, and pass data cleanly into the machine learning pipeline were all built on the foundations laid in these weeks. That connection — between the abstract exercises of this unit and the real system we would eventually build — is exactly the kind of link that [Dr. Bilal Ahmad](https://www.linkedin.com/in/drbilalphd/) builds into his courses deliberately, and one that I came to appreciate fully only once we were deep into the project itself.
