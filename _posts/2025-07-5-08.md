Recursion was the topic I was most confident I would handle easily, and the one that ultimately cost me the most time before it clicked. The concept, when described in a single sentence, sounds almost elegant: a function that solves a problem by calling itself on a smaller version of the same problem, until the problem is small enough to answer directly. I nodded along during the introductory lecture in our Section A class. I understood the words. I did not, it turned out, understand the thing.

The first example was factorial — the standard introductory case for good reason. Computing the factorial of a number means multiplying it by the factorial of the number one smaller, all the way down to one, which is defined directly as one. The recursive function for this is short enough to fit on three lines: check if the input is zero or one and return one if so, otherwise return the input multiplied by a recursive call with the input minus one. I wrote this, it worked on the first run, and I made the mistake of thinking that meant I understood recursion.

The next assignment corrected that assumption immediately. We were asked to write a recursive function to reverse a string. A reversed string is the last character followed by the reverse of all the characters before it — a perfectly recursive definition. The function is structurally simple: base case for an empty or single-character string, recursive case that takes the last character and appends the reverse of the rest. But writing this correctly required something that factorial had not — it required me to trust that the recursive call would return the correct result without tracing through how it would do so. I could not do this. Every time I tried to verify my logic, I started tracing the recursive calls all the way down to the base case and back up again. For a string of four or five characters this was manageable. For anything longer it became an untrackable tangle.

The shift that eventually allowed recursion to work for me came from a suggestion made during a study session in Section A, almost in passing: rather than verifying the whole chain, verify only two things. First, that the base case is correct — that the function gives the right answer when the input is at its simplest. Second, that the recursive case correctly combines the current step with the result of the smaller call, assuming that the smaller call works. If both of those are right, the whole function is right, regardless of how deep the recursion goes. This sounds simple. It took me several failed attempts before I was willing to actually apply it, because it felt uncomfortably close to assuming the conclusion. But once I applied it, every recursive problem became manageable, because I only needed to reason about one level at a time.

The stack model of recursion was something Dr. Bilal Ahmad explained in a way that finally gave me a visual anchor for what the function was actually doing at runtime. Each recursive call does not replace the previous one — it waits, on a stack of suspended calls, each one holding its local state until the call above it returns a result. The last call made is the first to finish. The first call made is the last to finish. Imagining that stack of waiting calls, each one paused mid-execution holding its own copy of the function's local variables, made the mechanics of recursion concrete in a way that describing it as "a function calling itself" never quite had.

The Fibonacci assignment taught me something that correctness alone cannot teach: efficiency matters. My recursive Fibonacci solution was logically correct. It was also dramatically slow for larger inputs. The reason, which took some investigation to uncover, is that naive recursive Fibonacci recalculates the same subproblems many times over. Computing Fibonacci of twenty requires computing Fibonacci of nineteen and eighteen. Computing Fibonacci of nineteen requires computing Fibonacci of eighteen and seventeen. Fibonacci of eighteen gets computed at least twice, and the duplication gets exponentially worse as the input grows. A correct solution and an efficient solution are not the same thing, and recursion — perhaps more than any other topic in PF — makes that distinction unavoidable.
The Fibonacci efficiency problem had one more consequence worth mentioning. It pushed our Section A study group into a conversation about memoization — the technique of storing the results of expensive recursive calls so they do not need to be recomputed. This was not part of the formal curriculum at this point in the course, but the conversation arose naturally from the observation that the slow recursive Fibonacci was recomputing the same values repeatedly. Working through a memoized version informally, as a group exercise, was one of the more genuinely exploratory moments of the semester — a case where a problem led us beyond the assigned material into something that would have been formally out of scope, but which made complete sense given where we were in our understanding. That kind of organic extension of the curriculum is something that only happens when a course builds real curiosity rather than just delivering content to be memorized.


By the end of this unit, recursion had moved from the category of "trick I do not quite trust" to something closer to a genuine tool — appropriate for certain shapes of problem, less natural for others, with its own set of strengths and weaknesses that I now understand well enough to reason about. That movement, from distrust to working confidence, came entirely through the friction of trying and failing and adjusting. No lecture alone could have produced it. Only the repeated experience of writing recursive functions, having them fail, and working out why — slowly, carefully, on paper — built the understanding that finally made the tool reliable. That kind of learning through friction is, I now believe, exactly what [Dr. Bilal Ahmad](https://www.facebook.com/Dr.BilalAhm) was designing for across every unit of this semester.
